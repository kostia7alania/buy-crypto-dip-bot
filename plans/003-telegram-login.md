# ExecPlan 003 — Telegram Login on the Dashboard

Backlog item #5 in `docs/14_PRODUCT_STRATEGY.md`: unify bot + web identity.
The `users` table already keys identity by `telegramUserId`; the bot writes it
on `/start`. This plan gives the web dashboard the same identity via the
official [Telegram Login Widget](https://core.telegram.org/widgets/login).

## Design

Verification lives in the **API** (it already holds `TELEGRAM_BOT_TOKEN` for
the runner); the Nuxt server stays a thin BFF that only manages the session
cookie. Flow:

1. Dashboard renders the Telegram Login Widget (needs the bot username,
   public runtime config `NUXT_PUBLIC_TELEGRAM_BOT_USERNAME`).
2. Widget calls `window.onTelegramAuth(user)` → BFF `POST /api/auth/telegram`.
3. BFF forwards the payload to API `POST /auth/telegram` (x-api-key attached
   server-side as usual).
4. API verifies the payload per the Telegram spec — HMAC-SHA256 over the
   sorted `key=value` data-check string with `SHA256(bot_token)` as key,
   timing-safe compare, `auth_date` freshness window (24 h) — then upserts
   the user by `telegramUserId` and writes a `USER_WEB_LOGIN` audit event.
5. BFF stores `{ user }` in an h3 sealed session cookie
   (`SESSION_SECRET`, HttpOnly). `GET /api/auth/me` and
   `POST /api/auth/logout` complete the session API.

## Components

- `apps/api/src/modules/auth/` — `auth.service.ts` (pure verification,
  exported for tests), `auth.route.ts` (valibot-validated `POST /telegram`),
  `auth.test.ts` (accept valid, reject bad hash / tampered field / stale
  `auth_date`). Mounted at `/auth` in `app.ts`.
- `apps/web/server/utils/session.ts` — shared `useAppSession` (h3
  `useSession`; `SESSION_SECRET` env, random per-boot fallback in dev).
- `apps/web/server/api/auth/{telegram.post,me.get,logout.post}.ts`.
- `apps/web/app/entities/user/` + `apps/web/app/widgets/telegram-login/` —
  session state + login/logout UI in the dashboard header.
- Env: `SESSION_SECRET` (generated by `scripts/vps-bootstrap.sh`),
  `NUXT_PUBLIC_TELEGRAM_BOT_USERNAME` (manual). The widget only works after
  `/setdomain` is configured in BotFather for the site domain.

## Non-goals (this iteration)

- No per-user data scoping yet — the dashboard stays single-tenant; login
  only establishes the shared identity multi-user will build on.
- No auth gate on dashboard routes; the API key remains the perimeter.

## Verification

- `pnpm check` (auth tests included).
- Manual: log in via the widget on the dashboard, see the username in the
  header, `users` row upserted, `USER_WEB_LOGIN` audit event; logout clears
  the cookie.
